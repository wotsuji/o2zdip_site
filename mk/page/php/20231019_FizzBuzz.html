FizzBuzzを書いてみる

FizzBuzzって再帰処理で実装したり、DI注入（依存性の注入）など、設計のベストプラクティスみたいなモノはあったりするのですが
運用保守開発的な実務していて、こう書いてあったら修正しやすいのになって感じの私なりのFizzBuzzにしてます。
再帰やDI注入はコードをよく読んで理解していく必要があって新規の時はいいのですが
小規模な修正をその場その場で繰り返している運用開発だと、やってる身としてはイチイチ仕様調査しないといけないのでツライ。
という及び腰な感じになっています。

◆実装コード
インデントが潰れちゃってるので全容は後述のGitHubのコードを参照してください。
// FizzBuzz
// 実行方法： php -f fizzbuzz.php

// FizzBuzzターゲット配列
$TARGET_ARRAY = [
[3, "Fizz"],
[5, "Buzz"]
];
// HIT無しの時の出力テキスト
$SKIP_TEXT = "-";
// 1～Limit
$LIMIT = 100;

$fizzbuzz = new FizzBuzz();
$output_text = $fizzbuzz->fizzBuzzExec($TARGET_ARRAY, $SKIP_TEXT, $LIMIT);
echo $output_text;

class FizzBuzz
{

public function fizzBuzzExec($target_array, $skip_text, $limit)
{
$output_text = "";
for ($i = 1; $i <= $limit; $i++) {
$output_line = $this->fizzBuzzOutputLine($target_array, $skip_text, $i);
$output_text .= $i . ":" . $output_line . "\n";
}
return $output_text;
}

private function fizzBuzzOutputLine($target_array, $skip_text, $check_int)
{
$output_line = $this->fizzBuzzOutputLineVals($target_array, $check_int);
if ($output_line == "") {
$output_line = $skip_text;
}
return $output_line;
}

private function fizzBuzzOutputLineVals($target_array, $check_int)
{
$output_line = "";
foreach ($target_array as $vals) {
if ($check_int % $vals[0] == 0) {
$output_line .= $vals[1];
}
}
return $output_line;
}
}

◆ちょっとだけ解説...
// FizzBuzzターゲット配列
$TARGET_ARRAY = [
[3, "Fizz"],
[5, "Buzz"]
];
なるべく設定を外に出して実行するクラスに渡すようにしています。
今回、配列で「3だとFizz」「5だとBuzz」をセットにしてforeachで回す形を取ってます。
この形にしておくと追加や変更があった時の修正箇所が見やすい。

// HIT無しの時の出力テキスト
$SKIP_TEXT = "-";
これは引数で渡すべきなのか迷うところ。クラス側で固定で持つのも良いかもしれないです。
あくまで主観ですが「ここは変えたい」と言われる予感があり実行側から変更できるようにしました。
運用の仕様変更でよくあるのは「全体の出力は仕様となっている」という旨を理解いただける事が多いのですが
軽微な部分の変更はコード修正して力技で実装をしないといけない事が多いです。
※フラグを追加で渡してクラス側の条件分岐が増えて双方向に依存が強まる・・・というのが私の経験上は実装が多いです。

◆まとめ
本当はPSRとかSolid原則やクリーンアーキテクチャに理解があって、組織として学んでいるのが理想と思いますが
PHP勉強会とかカンファレンスに参加する人が居る組織って、私の知ってる範囲では完全な少数派です。
再帰などで難しく書き過ぎず、かといってクラス側に依存し過ぎずのバランスに気を使う事が大事と思ってます。
運用に入って会った事の無い人（スキルに差がある人）が担当しても、そこまでストレスにならないコードの程度かと思います。
※あくまで個人の主観です。
